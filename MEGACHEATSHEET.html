Minimal Express server — every line explained

const express = require("express"); // load Express module from node_modules
const app     = express();         // create Express application instance
const PORT    = 3000;              // port number (or use process.env.PORT)

// Route: method + path + handler function
app.get("/", (req, res) => {
  res.send("Hello from Express!"); // sends plain text, status 200
});

app.get("/about", (req, res) => {
  res.send("About Page");
});

// app.listen MUST come AFTER all route definitions
app.listen(PORT, () => {
  console.log(`Server running on http://localhost:${PORT}`);
});
// listen() → starts HTTP server, binds to PORT
// callback fires once server is ready to accept connections

Key concepts from P01

Concept What it is / Why it matters
require("express") CommonJS module import. Loads Express from node_modules/ into the variable. Without this, you can't use any Express features.
express() Creates the app instance. All routes and middleware attach to this object via app.get(), app.use() etc.
app.get(path, handler) Defines a GET route. When a GET request hits path, Express calls handler(req, res).
req Request object — contains everything about the incoming HTTP request: headers, URL params, body, query string.
res Response object — methods to send a response back to the client: res.send(), res.json(), res.status().
res.send() Sends any type (string, Buffer, object). Sets Content-Type automatically.
res.json(data) Sends JSON response + sets Content-Type: application/json.
app.listen(port, cb) Starts server. Must be LAST — routes must be defined before server starts listening.
node app.js Run the app. Press Ctrl+C to stop. Must restart after code changes.

res.send vs res.json — when to use which

// res.send — plain text, HTML, or simple string
res.send("Hello World");        // Content-Type: text/html
res.send({ a: 1 });           // also works but use res.json for objects

// res.json — use for API responses (objects, arrays)
res.json([{ id: 1, name: "Apple" }]); // Content-Type: application/json
res.json({ error: "Not found" });

// Chaining status then json:
res.status(201).json({ id: 5 }); // 201 Created
res.status(404).json({ error: "Not found" });
res.status(204).send();         // No Content — send() with no arg
🌐 2 · REST CONCEPTS — What, Why, How

What is REST?

REST = Representational State Transfer. An architectural style (not a protocol) for designing networked APIs.

Key REST principles:
• Client-Server: Frontend and backend are separate, communicate via HTTP
• Stateless: Every request contains ALL info needed — server stores no session
• Uniform Interface: Resources identified by URLs; standard HTTP methods used
• Resources: Entities (books, users, foods) identified by URLs like /books/5

HTTP Methods → CRUD Mapping

Method CRUD SQL Success Status
GET Read SELECT 200 OK
POST Create INSERT 201 Created
PUT Update UPDATE 200 OK
DELETE Delete DELETE 204 No Content
Why different methods for same URL? GET /books reads all; POST /books creates one. Method + URL together define what action to take.

HTTP Status Codes — Complete List

Code Meaning Use case
200 OK GET, PUT success
201 Created POST creates resource
204 No Content DELETE success (no body)
400 Bad Request Validation fail, invalid id
401 Unauthorized Not logged in / no credentials
403 Forbidden Logged in but no permission
404 Not Found Resource doesn't exist in DB
500 Internal Server Error Unhandled DB/server crash

🍎 3 · IN-MEMORY CRUD — Practical 02 (Foods API)

Complete foods-api app.js (P02 pattern) — every line explained 

const express = require("express");
const app     = express();
const PORT    = 3000;

app.use(express.json());   // MUST be before routes — parses JSON bodies into req.body

let foods = [];              // in-memory "database" — resets on server restart
// Each item: { id: number, name: string, calories: number }

// ── CREATE (POST /foods) ────────────────────────────────────────────────
app.post("/foods", (req, res) => {
  const { name, calories } = req.body; // destructure from parsed JSON body

  if (!name || calories == null) {   // validation: both fields required
    // calories == null catches both null AND undefined (== not ===)
    return res.status(400).json({ message: "name and calories are required" });
  }

  const newFood = { id: Date.now(), name, calories };
  // Date.now() = milliseconds since epoch → unique id (enough for in-memory)
  // Shorthand: { name } is same as { name: name }

  foods.push(newFood);                // add to array
  res.status(201).json({ message: "Food created successfully.", food: newFood });
});

// ── READ ALL (GET /foods) with optional filter ──────────────────────────
app.get("/foods", (req, res) => {
  const { name } = req.query;       // req.query = ?name=App → { name: "App" }
  let results = foods;

  if (name) {
    results = foods.filter(f => f.name.includes(name));
    // .filter() returns new array of items where callback returns true
    // .includes() = partial string match (case-sensitive)
  }
  res.json({ message: `Found ${results.length} food(s)`, foods: results });
});

// ── UPDATE (PUT /foods/:id) ──────────────────────────────────────────────
app.put("/foods/:id", (req, res) => {
  const foodId = Number(req.params.id);  // URL param is a STRING → convert
  // Date.now() generated IDs are numbers, so we use Number() not parseInt()
  const { name, calories } = req.body;

  if (!name || calories == null) {
    return res.status(400).json({ message: "name and calories are required" });
  }

  const idx = foods.findIndex(f => f.id === foodId);
  // findIndex returns -1 if not found
  if (idx === -1) {
    return res.status(404).json({ message: `No food found with id ${foodId}` });
  }

  foods[idx] = { id: foodId, name, calories }; // replace entirely
  res.json({ message: "Updated", food: foods[idx] });
});

// ── DELETE (DELETE /foods/:id) ───────────────────────────────────────────
app.delete("/foods/:id", (req, res) => {
  const foodId = Number(req.params.id);
  const exists = foods.some(f => f.id === foodId);
  // .some() returns true if ANY item matches predicate
  if (!exists) {
    return res.status(404).json({ message: `No food with id ${foodId}` });
  }
  foods = foods.filter(f => f.id !== foodId); // keep all EXCEPT this one
  res.json({ message: `Food ${foodId} deleted` });
});

app.listen(PORT, () => console.log(`Running on port ${PORT}`));

req.body vs req.query vs req.params

Property Source Example URL / body How to read
req.params.id URL segment defined as :id /foods/17 Number(req.params.id)
req.query.name Query string after ? /foods?name=App req.query.name
req.body.name JSON/form body POST body JSON req.body.name (needs express.json())

Array methods used in P02

Method Purpose Returns
.push(item) Add to end New length
.filter(fn) Keep matching items New array
.findIndex(fn) Find position Index or -1
.some(fn) Check if any match Boolean
.includes(str) String contains Boolean

Why in-memory is NOT production-ready

Problem 1 — Not persistent: All data lost when server restarts. let foods = [] is just a JS variable in RAM.
Problem 2 — Not scalable: If 2 server instances run, each has its own array — they get out of sync.
Problem 3 — No queries: Must filter in JS — slow for large datasets.
Solution → MSSQL database (P03+): Persistent, concurrent, queryable.

mssql package — how it works

const sql = require("mssql");
// npm install mssql to install

// ── CONNECT ───────────────────────────────────────
const connection = await sql.connect(dbConfig);
// Opens a connection using credentials in dbConfig
// Returns a connection object

// ── CREATE REQUEST ────────────────────────────────
const request = connection.request();
// Creates a SQL request object on this connection

// ── BIND PARAMETERS (parameterized query) ─────────
request.input("id", bookId);
// "id" = name used as @id in SQL, bookId = value
// CRITICAL: prevents SQL injection — treats value as DATA not code

// ── EXECUTE QUERY ─────────────────────────────────
const result = await request.query("SELECT * FROM Books WHERE id = @id");

// ── ACCESS RESULTS ────────────────────────────────
result.recordset        // array of row objects
result.recordset[0]     // first row (single record)
result.recordset.length // how many rows returned
result.rowsAffected[0]  // # rows changed (UPDATE/DELETE)

// ── CLOSE CONNECTION ─────────────────────────────
await connection.close();
// MUST call in finally block — releases DB resources

result object structure

{
  recordsets: [ [/* 1st result set */], [/* 2nd */] ],
  recordset:  [/* 1st result set — shortcut */],
  // recordset is array of plain objects:
  // [{ id: 1, title: "Dune", author: "Herbert" }, ...]
  // Column names in SELECT → object keys
  // Column aliases (AS) → object keys
  output:      {},          // output parameters (stored procs)
  rowsAffected: [1]         // one entry per SQL statement
}

// Checking if record exists:
if (result.recordset.length === 0) return null;

// Checking if update/delete found the row:
return result.rowsAffected[0] > 0; // true if found

Why MVC? (P03 problems solved)

P03 Problem MVC Solution
All DB logic inside route handlers — messy DB logic extracted into Model files
Hard to reuse DB functions Model functions callable from anywhere
Can't test DB logic separately Model is independently testable
No validation → 500 on bad input Joi middleware validates before controller
Passwords hardcoded in dbConfig dotenv → credentials in .env
MVC = Model + View + Controller
Model — only database logic (queries, connections)
View — HTML/JS frontend (P05)
Controller — request/response handling, calls Model
Middleware — validation, auth, sits between route and controller

Key rule: Controller NEVER talks to DB directly. Model NEVER touches req/res. This separation means you can change how DB works without touching routes, and vice versa.

const express = require("express");
// Load Express from node_modules — enables app.get(), app.use(), app.listen() etc.

const sql = require("mssql");
// Load mssql — needed here ONLY for sql.close() in graceful shutdown
// (actual DB queries happen inside models)

const dotenv = require("dotenv");
// Load dotenv package — used to read .env file into process.env

const path = require("path");
// Built-in Node.js module — builds file paths safely across Windows/Mac/Linux

dotenv.config();
// MUST be called BEFORE accessing process.env.ANY_VARIABLE
// Reads .env file line by line: PORT=3000 → process.env.PORT = "3000"
// If called too late, process.env variables will be undefined

// ── IMPORT CONTROLLERS ─────────────────────────────────────────────────────
const bookController = require("./controllers/bookController");
// "./" = relative to current file (app.js is in project root)
// Returns what bookController.js assigned to module.exports

const userController = require("./controllers/userController");
// P06: added when user routes are needed

// ── IMPORT MIDDLEWARE ───────────────────────────────────────────────────────
const { validateBook, validateBookId } = require("./middlewares/bookValidation");
// Destructuring: extract named exports from the module
// Same as: const bv = require(...); const validateBook = bv.validateBook;

// ── APP SETUP ───────────────────────────────────────────────────────────────
const app  = express();
const port = process.env.PORT || 3000;
// process.env.PORT = value from .env file (string "3000")
// || 3000 = fallback if PORT not set in .env

// ── BODY PARSING MIDDLEWARE ─────────────────────────────────────────────────
app.use(express.json());
// Parses incoming JSON request bodies
// When client sends: Content-Type: application/json + JSON body string
// Express parses it into a JS object and puts it in req.body
// WITHOUT this: req.body would be undefined for JSON requests

app.use(express.urlencoded({ extended: true }));
// Parses URL-encoded form data (application/x-www-form-urlencoded)
// When HTML form submits: name=Alex&id=5 → { name: "Alex", id: "5" }
// extended: true = uses qs library (handles nested objects)
// extended: false = uses querystring library (simpler)

// ── STATIC FILES (P05) ───────────────────────────────────────────────────────
app.use(express.static(path.join(__dirname, "public")));
// __dirname = absolute path of current file, e.g. "C:/projects/books-api-mvc-db"
// path.join(__dirname, "public") = "C:/projects/books-api-mvc-db/public"
// express.static serves any file in that folder at the root URL
// GET /index.html → serves public/index.html
// GET /script.js  → serves public/script.js
// Place BEFORE API routes so static files are served without hitting API logic

// ── BOOK ROUTES ─────────────────────────────────────────────────────────────
app.get("/books",       bookController.getAllBooks);
// No middleware needed — reading all books doesn't need validation

app.get("/books/:id",   validateBookId, bookController.getBookById);
// validateBookId runs FIRST → if id invalid, sends 400 and stops
// If id valid, calls next() → bookController.getBookById runs

app.post("/books",      validateBook, bookController.createBook);
// validateBook checks req.body has valid title+author before createBook runs

app.put("/books/:id",   validateBookId, validateBook, bookController.updateBook);
// TWO middlewares: first validate id, then validate body, then update
// Express chains them in order — each must call next() to continue

app.delete("/books/:id", validateBookId, bookController.deleteBook);

// ── USER ROUTES (P06) ───────────────────────────────────────────────────────
app.post("/users",            userController.createUser);
app.get("/users",             userController.getAllUsers);
app.get("/users/search",      userController.searchUsers);
// IMPORTANT: /users/search MUST be defined BEFORE /users/:id
// Express matches routes in order — if :id is first, "search" would match it
// as id="search" and go to getUserById instead!

app.get("/users/with-books", userController.getUsersWithBooks);
app.get("/users/:id",         userController.getUserById);
app.put("/users/:id",         userController.updateUser);
app.delete("/users/:id",      userController.deleteUser);

// ── START SERVER ─────────────────────────────────────────────────────────────
app.listen(port, () => {
  console.log(`Server running on port ${port}`);
});
// Binds server to port. Callback fires when server is ready.
// node app.js → this is what starts everything

// ── GRACEFUL SHUTDOWN ────────────────────────────────────────────────────────
process.on("SIGINT", async () => {
  console.log("Server is gracefully shutting down");
  await sql.close(); // close mssql connection pool
  console.log("Database connections closed");
  process.exit(0);   // 0 = clean exit, 1 = error exit
});
// SIGINT = signal sent by Ctrl+C
// Without this: DB connections leak — eventually server can't open new ones
// process.on() registers an event listener on the Node.js process object

🗄️ 8 · bookModel.js — Complete with All 5 CRUD Functions 
const sql      = require("mssql");       // mssql package for SQL Server
const dbConfig = require("../dbConfig"); // "../" = up one level from models/ to root

// ═══════════════════════════════════════════════════════════════════
// TEMPLATE: try/catch/finally pattern used in EVERY model function
// - let connection declared outside try so finally can reference it
// - try: connect, build request, bind inputs, execute query, return result
// - catch: log + re-throw (controller's catch sends the 500 response)
// - finally: ALWAYS close connection regardless of success/failure
// ═══════════════════════════════════════════════════════════════════

// ── GET ALL ──────────────────────────────────────────────────────────────
async function getAllBooks() {
  let connection;                           // declared outside so finally block can access
  try {
    connection = await sql.connect(dbConfig); // open DB connection
    const query  = "SELECT id, title, author FROM Books";
    const result = await connection.request().query(query);
    // .request() creates SQL request object
    // .query(sql) executes it — returns result object
    return result.recordset;               // array of book objects
  } catch (error) {
    console.error("Database error:", error);
    throw error;                            // re-throw → controller catch sends 500
  } finally {
    if (connection) {
      try { await connection.close(); }    // always close — prevents connection leak
      catch (err) { console.error("Error closing:", err); }
    }
  }
}

// ── GET BY ID (parameterized) ─────────────────────────────────────────────
async function getBookById(id) {           // id passed in from controller
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query   = "SELECT id, title, author FROM Books WHERE id = @id";
    // @id = NAMED PARAMETER placeholder — mssql replaces it safely
    const request = connection.request();
    request.input("id", id);              // bind @id → value of id param
    // .input("paramName", value) — "paramName" matches @paramName in SQL
    // Value is treated as DATA not SQL code → prevents SQL injection
    const result = await request.query(query);
    if (result.recordset.length === 0) return null; // not found → return null
    return result.recordset[0];           // single row object
  } catch (error) {
    console.error("Database error:", error); throw error;
  } finally {
    if (connection) { try { await connection.close(); } catch(e){} }
  }
}

// ── CREATE (INSERT + SCOPE_IDENTITY) ──────────────────────────────────────
async function createBook(bookData) {      // bookData = req.body from controller
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query = `
      INSERT INTO Books (title, author) VALUES (@title, @author);
      SELECT SCOPE_IDENTITY() AS id;
    `;
    // SCOPE_IDENTITY() = returns the auto-generated id of the just-inserted row
    // Two statements in one query: INSERT first, then SELECT the new id
    const request = connection.request();
    request.input("title",  bookData.title);
    request.input("author", bookData.author);
    const result    = await request.query(query);
    const newBookId = result.recordset[0].id; // extracted new auto-generated ID
    return await getBookById(newBookId);       // fetch and return full book object
  } catch (error) {
    console.error("Database error:", error); throw error;
  } finally {
    if (connection) { try { await connection.close(); } catch(e){} }
  }
}

// ── UPDATE ────────────────────────────────────────────────────────────────
async function updateBook(id, bookData) {
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query = "UPDATE Books SET title = @title, author = @author WHERE id = @id";
    const request = connection.request();
    request.input("id",     id);
    request.input("title",  bookData.title);
    request.input("author", bookData.author);
    await request.query(query);             // no return needed from UPDATE
    return await getBookById(id);           // re-fetch updated book to return to client
  } catch (error) {
    console.error("Database error:", error); throw error;
  } finally {
    if (connection) { try { await connection.close(); } catch(e){} }
  }
}

// ── DELETE ────────────────────────────────────────────────────────────────
async function deleteBook(id) {
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query   = "DELETE FROM Books WHERE id = @id";
    const request = connection.request();
    request.input("id", id);
    const result  = await request.query(query);

const Joi = require("joi");
// npm install joi
// Joi = schema-based validation library
// You define what valid data LOOKS LIKE (schema), Joi checks actual data against it

// ── SCHEMA DEFINITION ────────────────────────────────────────────────────
const bookSchema = Joi.object({
  // Joi.object() = validates a JS object (matches req.body shape)

  title: Joi.string()         // must be a JavaScript string type
    .min(1)                   // at least 1 character long
    .max(50)                  // max 50 chars — matches VARCHAR(50) in DB
    .required()               // key must be present in req.body
    .messages({               // override default Joi error messages
      "string.base":  "Title must be a string",         // non-string given
      "string.empty": "Title cannot be empty",          // empty string ""
      "string.min":   "Title must be at least 1 character long",
      "string.max":   "Title cannot exceed 50 characters",
      "any.required": "Title is required",              // key missing entirely
    }),

  author: Joi.string().min(1).max(50).required().messages({
    "string.base":  "Author must be a string",
    "string.empty": "Author cannot be empty",
    "string.min":   "Author must be at least 1 character long",
    "string.max":   "Author cannot exceed 50 characters",
    "any.required": "Author is required",
  }),
});

// ── validateBook MIDDLEWARE ───────────────────────────────────────────────
function validateBook(req, res, next) {
  // MUST have 3 params: req, res, next — this is what makes it "middleware"
  // Express identifies it as middleware by the 3-arg signature

  const { error } = bookSchema.validate(req.body, { abortEarly: false });
  // .validate(data, options) → returns { error, value }
  // We destructure only 'error' — if validation passed, error is undefined
  //
  // abortEarly: false → collect ALL validation errors, not just the first
  // abortEarly: true (default) → stop at first error, return only that one
  // abortEarly: false is better UX — user sees all problems at once

  if (error) {
    // error.details = array of validation error objects
    // Each has: { message: "...", path: ["title"], type: "string.min" }
    // We extract just the message strings and join them
    const errorMessage = error.details
      .map(detail => detail.message) // extract message from each error
      .join(", ");                     // combine: "Title required, Author required"
    return res.status(400).json({ error: errorMessage });
    // return stops execution — controller never runs for invalid data
  }

  next(); // validation PASSED → call next(), which runs the controller function
}

// ── validateBookId MIDDLEWARE ─────────────────────────────────────────────
function validateBookId(req, res, next) {
  const id = parseInt(req.params.id);
  // req.params.id is always a STRING from URL
  // parseInt("5") → 5; parseInt("abc") → NaN; parseInt("5.9") → 5

  if (isNaN(id) || id <= 0) {
    // isNaN(NaN) → true (not a number)
    // id <= 0: negative numbers or 0 are invalid DB ids
    return res.status(400).json({ error: "Invalid book ID. ID must be a positive number" });
  }
  next(); // id is valid → pass to controller
}

async function searchUsers(searchTerm) {
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query = `
      SELECT * FROM Users
      WHERE username LIKE '%' + @searchTerm + '%'
         OR email    LIKE '%' + @searchTerm + '%'
    `;
    // LIKE '%' + @searchTerm + '%':
    //   % = wildcard matching any characters
    //   'user' matches 'user1', 'myuser', 'superuser'
    //   Concatenation done in SQL: '%' + value + '%'
    // OR: match in username OR email — both checked
    // @searchTerm = parameterized — prevents SQL injection in LIKE too!
    const request = connection.request();
    request.input("searchTerm", sql.NVarChar, searchTerm);
    // Explicitly typed as NVarChar (Unicode string) — recommended for LIKE queries

// ── GET USERS WITH BOOKS (LEFT JOIN + grouping) ────────────────────────────
async function getUsersWithBooks() {
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query = `
      SELECT u.id    AS user_id, u.username, u.email,
             b.id    AS book_id, b.title,    b.author
      FROM   Users u
      LEFT JOIN UserBooks ub ON ub.user_id = u.id
      LEFT JOIN Books     b  ON ub.book_id = b.id
      ORDER BY u.username;
    `;
    // LEFT JOIN Users → UserBooks: keeps users with 0 books (book cols = NULL)
    // LEFT JOIN UserBooks → Books: gets book details for each UserBooks row
    // Without LEFT JOIN (using INNER JOIN): users with 0 books would be excluded
    // AS aliases: u.id → user_id, b.id → book_id (avoid column name collision)
    // Result: flat rows, one per user-book combo
    //   user_id | username | email             | book_id | title
    //   1       | user1    | user1@example.com | 1       | Dune
    //   1       | user1    | user1@example.com | 2       | LOTR
    //   2       | user2    | user2@example.com | NULL    | NULL  ← user with 0 books

    const result = await connection.request().query(query);

    const usersWithBooks = {};  // keyed by user_id for O(1) lookup
    for (const row of result.recordset) {
      const userId = row.user_id;
      if (!usersWithBooks[userId]) {
        // First row for this user — initialize user object
        usersWithBooks[userId] = {
          id:       userId,
          username: row.username,
          email:    row.email,
          books:    [],   // empty array — books added below
        };
      }
      if (row.book_id !== null) {  // null = user has no books (LEFT JOIN)
        usersWithBooks[userId].books.push({
          id:     row.book_id,
          title:  row.title,
          author: row.author,
        });
      }
    }
    return Object.values(usersWithBooks);
    // Object.values() converts { 1: {...}, 2: {...} } → [{...}, {...}]
    // Returns an array of user objects, each with nested books array
  } catch (err) { console.error("Error in getUsersWithBooks:", err); throw err; }
  finally { if (connection) { try { await connection.close(); } catch(e){} } }
}

🔗 14 · MANY-TO-MANY RELATIONSHIPS (P06)

M:N Relationship Theory

Many-to-Many (M:N): A User can have many Books. A Book can be owned by many Users. Neither side can store the other's FK without duplication.
Rule: For each M:N relationship, create a JOIN TABLE containing the PKs of both entities as FKs. The composite PK = (FK1 + FK2).
Pattern: S(a1, b1, optional_attributes)
where a1 = FK to table A, b1 = FK to table B
Underlined = composite primary key

SQL Schema — all 3 tables

-- 1. Main table A
CREATE TABLE Users (
  id       INT PRIMARY KEY IDENTITY,
  username VARCHAR(50)  NOT NULL UNIQUE,
  email    VARCHAR(100) NOT NULL UNIQUE
);

-- 2. Main table B
CREATE TABLE Books (
  id     INT IDENTITY(1,1) PRIMARY KEY,
  title  VARCHAR(50) NOT NULL UNIQUE,
  author VARCHAR(50) NOT NULL
);

-- 3. Join/Bridge/Associative table
CREATE TABLE UserBooks (
  id      INT PRIMARY KEY IDENTITY,
  user_id INT FOREIGN KEY REFERENCES Users(id),
  -- user_id FK: links to Users table primary key
  book_id INT FOREIGN KEY REFERENCES Books(id)
  -- book_id FK: links to Books table primary key
);

-- Insert relationship: User 1 has Books 1 and 2
INSERT INTO UserBooks (user_id, book_id) VALUES (1,1),(1,2),(2,3);

LEFT JOIN vs INNER JOIN

JOIN Type Result When to use
INNER JOIN Only rows with matches in BOTH tables When you only want users WITH books
LEFT JOIN ALL rows from left table + matching from right (NULL if no match) When you want ALL users, even those with 0 books
RIGHT JOIN ALL rows from right table + matching from left All books even without users
P06 uses LEFT JOIN so users with 0 books still appear (book_id = null). The JS code checks if (row.book_id !== null) before pushing to the books array. 
-- Result from LEFT JOIN (flat rows):
user_id | username | email             | book_id | title
--------|----------|-------------------|---------|------
1       | user1    | user1@example.com | 1       | Dune
1       | user1    | user1@example.com | 2       | LOTR
2       | user2    | user2@example.com | NULL    | NULL

async / await — deep explanation

// async makes function return a Promise
async function example() {
  // everything here becomes a promise chain
}

// await pauses THIS function until Promise resolves
const result = await sql.connect(dbConfig);
// Without await: would get a Promise object, not the connection
// With await: waits for connection to open, then gives us the object
// Node event loop is NOT blocked — other requests run during the wait

// Promise states:
// Pending    → operation in progress
// Fulfilled  → operation succeeded (await gives resolved value)
// Rejected   → operation failed (await throws error → caught by catch)
Why async for DB ops? DB queries take time (network I/O). If synchronous, the entire Node.js server would freeze waiting. Async = other requests keep running during DB wait.

try / catch / finally — pattern deep dive

let connection;        // ← outside so finally can see it
try {
  connection = ...;   // ← opens connection
  const r = await ...; // ← executes query
  return r.recordset; // ← happy path
} catch (error) {
  console.error(error);
  throw error;         // re-throw → controller sends 500
} finally {
  if (connection) {   // guards against connect failing
    try { await connection.close(); }
    catch (e) {}      // ignore close errors
  }
}
// finally ALWAYS runs — success OR failure
// Without finally: if query throws, connection never closes → LEAK
// Connection leak → eventually server can't make new DB connections

SQL Injection — why parameterized queries

// ❌ UNSAFE — string concatenation
const id    = "1; DROP TABLE Books;--"; // attacker input
const query = `SELECT * FROM Books WHERE id = ${id}`;
// Executes: SELECT * FROM Books WHERE id = 1; DROP TABLE Books;--
// Result: Books table DELETED!

// ✅ SAFE — parameterized query
const query   = "SELECT * FROM Books WHERE id = @id";
const request = connection.request();
request.input("id", "1; DROP TABLE Books;--");
// mssql sends SQL and parameters SEPARATELY to SQL Server
// SQL Server treats value as DATA, not as executable SQL code
// Attacker input is harmless — just becomes a string value
Rule: ALWAYS use request.input("name", value) and @name in query for ANY user-provided data.

SQL clauses explained

Clause Purpose
IDENTITY(1,1) Auto-increment: starts at 1, increments by 1. SQL Server generates it automatically on each INSERT.
PRIMARY KEY Uniquely identifies each row. Auto-indexed. Cannot be NULL.
FOREIGN KEY REFERENCES T(col) Enforces referential integrity — value must exist in referenced table.
NOT NULL Field is required — INSERT fails if omitted.
UNIQUE No duplicate values allowed across rows.
VARCHAR(n) Variable-length string up to n chars.
SCOPE_IDENTITY() Returns auto-generated ID of the most recent INSERT in current scope.
LIKE '%x%' Pattern match: % = zero or more characters. Contains 'x' anywhere in string.
LEFT JOIN All rows from left table + matching from right. NULL if no match.
AS alias Rename column in results: u.id AS user_id → accessible as row.user_id.
ORDER BY Sort results. ASC (default) or DESC.

THEORY QUESTIONS (Q1 type)

Q: What is async/await and why is it used for database operations in Node.js?
The async keyword makes a function return a Promise and enables the use of await inside it. The await keyword pauses the execution of the surrounding async function until the awaited Promise resolves, then resumes with the resolved value. Crucially, this does NOT block the Node.js event loop — while the async function waits for the database to respond, Node.js continues handling other incoming requests.

For database operations specifically, each query involves network I/O which can take tens to hundreds of milliseconds. If these were synchronous (blocking), the entire Node.js server would freeze for every single request, making it unable to serve any other user. Using async/await allows hundreds of concurrent users to have their requests processed efficiently on Node.js's single thread.
Q: Why is dbConfig required in bookModel.js? What does it contain and why is it needed?
const dbConfig = require("../dbConfig") is needed because the line await sql.connect(dbConfig) requires a configuration object containing the database connection details — specifically: user (SQL Server login username), password, server (hostname), database (database name), trustServerCertificate, and options.port. Without importing dbConfig, the model has no credentials and sql.connect() would not know which SQL Server to connect to, what credentials to use, or which database to open. The model would fail to establish any connection.
Q: What is the purpose of the try/catch/finally block in model functions? Why is 'finally' specifically needed?
The try block contains the core database logic. If any await call throws an error (connection failure, query error, etc.), the catch block handles it by logging the error and re-throwing it so the controller can send a 500 response to the client.

The finally block is critical because it runs regardless of whether the try block succeeded or the catch block ran. Its purpose is to always close the database connection (connection.close()). Without finally, if an error occurs inside the try block before the close() call, the connection is never closed — it "leaks." Over time, leaked connections accumulate until the database's connection limit is reached, and the server can no longer open new connections. The finally block guarantees cleanup in all scenarios.
Q: What is dotenv and why is it used? What happens if dotenv.config() is not called?
dotenv is an npm package that reads a .env file and loads each KEY=VALUE line into process.env. This allows storing sensitive configuration (database passwords, ports, API keys) outside of source code, so they're never accidentally committed to version control.

dotenv.config() must be called early in app.js — before any code that accesses process.env.DB_USER etc. If not called, all process.env.DB_* values would be undefined, causing dbConfig.js to export undefined values, and sql.connect() would fail with authentication errors because no credentials were provided.
Q: What is the purpose of SIGINT and graceful shutdown? What happens without it?
SIGINT (Signal Interrupt) is the signal sent to a process when the user presses Ctrl+C in the terminal. process.on("SIGINT", async () => { ... }) registers a handler that runs before the process exits.

The graceful shutdown calls sql.close() to close the mssql connection pool, releasing all active database connections. Without this, when the Node.js process is killed, any open database connections are abruptly terminated — connections may be left in an inconsistent state on the SQL Server side, and connection pool resources are not properly released. This can cause issues like "too many connections" errors when restarting the server repeatedly.
Q: Explain the role of express.json() and express.urlencoded(). What is req.body without them?
express.json() is middleware that parses incoming request bodies with Content-Type: application/json. When a client sends a JSON string in the body (e.g., from fetch() with JSON.stringify()), this middleware converts it to a JavaScript object and makes it available as req.body.

express.urlencoded() does the same for application/x-www-form-urlencoded data, which is what HTML forms submit by default.

Without these middlewares, req.body would be undefined — Express does not parse request bodies automatically. Any code trying to access req.body.title would get a TypeError trying to read property 'title' of undefined.
Q: Why must specific routes like /users/search be defined BEFORE /users/:id in app.js?
Express matches routes in the order they are defined. :id is a URL parameter wildcard — it matches ANY string segment. If GET /users/:id is defined first, a request to GET /users/search would match it with req.params.id = "search", calling getUserById with id = "search" instead of searchUsers. By placing /users/search before /users/:id, Express matches the literal string "search" first, calling the correct handler. The :id route only fires if no earlier route matched.
Q: What is SQL injection and how do parameterized queries prevent it?
SQL injection is an attack where malicious SQL code is embedded in user input data. For example, if an id field contains "1; DROP TABLE Books;--" and is concatenated directly into a SQL string, the database executes both the SELECT and the DROP TABLE commands, potentially destroying data or exposing sensitive information.

Parameterized queries prevent this by keeping SQL code and user data strictly separate. The query contains a placeholder (@id), and request.input("id", value) sends the value as a separate parameter. The database driver (mssql) ensures the value is treated as pure data — it can never be interpreted as SQL code, regardless of what characters it contains. This makes injection attacks impossible.
Q: What does module.exports do, and why is it needed?
In Node.js, each file is a separate module with its own scope — variables and functions defined in one file are not automatically available in others. module.exports is the mechanism to explicitly share values (functions, objects, classes) between files.

What you assign to module.exports is what the importing file gets when it calls require("./path/to/file"). For example, assigning module.exports = { getAllBooks, createBook } in bookModel.js means const bookModel = require("../models/bookModel") in bookController.js gives access to bookModel.getAllBooks() and bookModel.createBook(). Without module.exports, the MVC pattern cannot work — controllers couldn't call model functions.
Q: Explain MVC architecture. What goes in each layer and why is the separation important?
Model contains all database interaction logic: SQL queries, connection management, parameterized queries, and returning raw data. It has no knowledge of HTTP requests or responses.

View is the frontend layer — HTML, CSS, and client-side JavaScript. In P05, this is the public/ folder served by express.static(). It makes HTTP requests to the API and renders responses for the user.

Controller handles HTTP requests and responses: extracts data from req.params, req.body, req.query, calls Model functions, checks results, and sends responses with appropriate status codes. It never contains SQL queries.

The separation matters because: (1) Database logic can be changed without touching routes, (2) Model functions can be reused across multiple controllers, (3) Each layer can be tested independently, (4) Code is more readable since each file has one clear responsibility.

MIDDLEWARE QUESTIONS (Q2 type)

Q: What is middleware? How is it different from a route handler? How do you use a middleware in a route?
Middleware is a function with the signature (req, res, next) that sits between an incoming HTTP request and the final route handler. It has access to the request and response objects and can: execute code, modify req/res, end the request-response cycle, or call next() to pass control to the next function.

A regular route handler is (req, res) — no next. A middleware is (req, res, next) — it can chain.

To use middleware in a route, list it as an argument before the controller function: app.post("/books", validateBook, bookController.createBook). Express calls validateBook first. If it calls next(), then createBook runs. If it sends a response (e.g., 400), createBook never runs. The order of arguments is the order of execution.
Q: What does abortEarly: false do in Joi validation? Why is it better than the default?
abortEarly: false is an option passed to bookSchema.validate(req.body, { abortEarly: false }). It instructs Joi to continue validating all fields even after the first error is found, collecting every validation failure into the error.details array.

The default behaviour (abortEarly: true) stops validation at the first error, returning only that one error message. This means a user must fix one error, resubmit, discover the next error, resubmit again, and so on.

With abortEarly: false, the client receives all validation errors simultaneously (e.g., "Title is required, Author cannot exceed 50 characters") in a single response, allowing the user or frontend to fix all problems at once. This is significantly more user-friendly for API consumers.
Q: What does validateBookId middleware check and why?
validateBookId extracts the :id URL parameter with parseInt(req.params.id) and checks two conditions: (1) isNaN(id) — is the parsed result not a number? This catches non-numeric inputs like "abc". (2) id <= 0 — negative numbers and zero are not valid database IDs since SQL Server IDENTITY starts at 1.

If either condition fails, it returns a 400 Bad Request immediately, stopping the request before it reaches the controller and model. This prevents the model from receiving invalid input that would cause a meaningless database query, and gives the client a clear, specific error message rather than a cryptic 500 server error.
Q: Write a Joi validation schema for a Food item with fields: name (string, required, 1-50 chars) and calories (number, required, min 0). 
const Joi = require("joi");

const foodSchema = Joi.object({
  name: Joi.string().min(1).max(50).required().messages({
    "string.base":  "Name must be a string",
    "string.empty": "Name cannot be empty",
    "string.min":   "Name must be at least 1 character long",
    "string.max":   "Name cannot exceed 50 characters",
    "any.required": "Name is required",
  }),
  calories: Joi.number().min(0).required().messages({
    "number.base":  "Calories must be a number",
    "number.min":   "Calories must be at least 0",
    "any.required": "Calories is required",
  }),
});

function validateFood(req, res, next) {
  const { error } = foodSchema.validate(req.body, { abortEarly: false });
  if (error) {
    const msg = error.details.map(d => d.message).join(", ");
    return res.status(400).json({ error: msg });
  }
  next();
}
module.exports = { validateFood };

MODEL QUESTIONS (Q3 type)

Q: Write a model function getAllFoods() that retrieves all records from a Foods table in MSSQL. 
const sql      = require("mssql");
const dbConfig = require("../dbConfig");

async function getAllFoods() {
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query  = "SELECT id, name, calories FROM Foods";
    const result = await connection.request().query(query);
    return result.recordset;
  } catch (error) {
    console.error("Database error:", error);
    throw error;
  } finally {
    if (connection) {
      try { await connection.close(); }
      catch (err) { console.error("Error closing connection:", err); }
    }
  }
}
module.exports = { getAllFoods };
Q: Write a model function getFoodById(id) that returns a single food or null if not found. 
async function getFoodById(id) {
  let connection;
  try {
    connection = await sql.connect(dbConfig);
    const query   = "SELECT id, name, calories FROM Foods WHERE id = @id";
    const request = connection.request();
    request.input("id", id);   // parameterized — @id bound to id value
    const result  = await request.query(query);
    if (result.recordset.length === 0) return null; // not found
    return result.recordset[0];  // single food object
  } catch (error) {
    console.error("Database error:", error); throw error;
  } finally {
    if (connection) { try { await connection.close(); } catch(e){} }
  }
}

Q: What is SCOPE_IDENTITY() and why is it used after INSERT?
When you INSERT a row into a table with an IDENTITY column (auto-incrementing primary key), SQL Server generates the new ID automatically — but the INSERT statement itself doesn't return it directly. SELECT SCOPE_IDENTITY() AS id retrieves the most recently generated IDENTITY value within the current scope (current connection and current stored procedure).

It is used in the model's createBook/createUser functions right after INSERT so we know the new record's ID. We then pass this ID to getBookById(newId) to fetch and return the complete newly created record to the client. Without SCOPE_IDENTITY(), we wouldn't know which ID was assigned, and couldn't confirm the creation succeeded or return the complete resource.
Q: Describe the LEFT JOIN used in getUsersWithBooks. What would change if you used INNER JOIN instead?
The LEFT JOIN joins Users to UserBooks on ub.user_id = u.id, then UserBooks to Books on ub.book_id = b.id. A LEFT JOIN keeps ALL rows from the left table (Users) even when there are no matching rows in the right table (UserBooks). For a user with no books, book_id, title, and author columns will be NULL in those rows.

The JavaScript code handles this with if (row.book_id !== null) — only adding a book to the array when one actually exists, leaving the books array empty for users with no books.

With INNER JOIN, only users who have at least one entry in UserBooks would appear in the results. Users with 0 books would be completely excluded from the response, which would be incorrect behavior if we want to show all users regardless of whether they have books.

⚠️ Common Mistakes to Avoid

1. Forgetting dotenv.config()
Must be FIRST in app.js before any process.env access. Otherwise all env vars are undefined.
2. await without async
You can't use await in a non-async function. Always mark function as async if it awaits anything.
3. Missing express.json()
Without app.use(express.json()), req.body is undefined for JSON POST requests.
4. /users/:id before /users/search
:id matches ANY segment. search-style routes must come BEFORE parameterized routes.
5. String concatenation in SQL
NEVER use template literals to build SQL with user input. Always use request.input() + @param.
6. No finally block
If query throws, connection never closes → leak. Always close in finally.
7. Not re-throwing in catch
Model must throw error so controller catch can send 500. Without throw, controller gets undefined.
8. Missing parseInt()
req.params.id is always a string. SQL expects an integer. Use parseInt() in controller.
9. Committing .env to git
.env must be in .gitignore. Once in git history, credentials are compromised forever.
